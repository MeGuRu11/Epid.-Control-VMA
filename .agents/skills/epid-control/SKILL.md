# Epid Control — Проектный скилл

## Перед началом любой задачи — выполни этот чеклист:

1. ☐ Прочитай `docs/context.md` — пойми текущее состояние и роадмап.
2. ☐ Прочитай `docs/progress_report.md` — узнай, что делалось недавно.
3. ☐ Определи затронутые слои архитектуры (UI, Application, Domain, Infrastructure).
4. ☐ Если задача связана с другим скиллом (напр. database-schema-designer) — прочитай его SKILL.md.

## Архитектура (Clean Architecture / DDD)

```
┌──────────────────────────────────────┐
│   UI Layer (PySide6 виджеты)         │
│   Формы, диалоги, таблицы            │
│   ПРАВИЛО: никакой бизнес-логики     │
└──────────────┬───────────────────────┘
               │ Вызовы сервисов
┌──────────────▼───────────────────────┐
│   Application Layer                  │
│   Use cases, сервисы приложения      │
│   ПРАВИЛО: оркестрирует, не содержит │
│   бизнес-правил напрямую             │
└──────────────┬───────────────────────┘
               │ Вызовы доменных объектов
┌──────────────▼───────────────────────┐
│   Domain Layer                       │
│   Сущности, value objects            │
│   ПРАВИЛО: НЕТ зависимостей вообще   │
│   Чистая бизнес-логика и валидация   │
└──────────────┬───────────────────────┘
               │ Реализация интерфейсов
┌──────────────▼───────────────────────┐
│   Infrastructure Layer               │
│   SQLAlchemy модели, репозитории     │
│   Alembic миграции, файловый I/O     │
│   ПРАВИЛО: реализует протоколы Domain│
└──────────────────────────────────────┘
```

### ЗАПРЕЩЁННЫЕ зависимости (нарушение = ошибка архитектуры):
- UI ← Infrastructure (UI не должен знать о SQLAlchemy)
- Domain ← PySide6 (Domain не должен знать о GUI)
- Domain ← SQLAlchemy (Domain не должен знать о ORM)

## Обязательный цикл после КАЖДОГО изменения кода

```bash
# Шаг 1: Линтер (ОБЯЗАТЕЛЬНО)
ruff check app tests
# Если ошибки — исправь и повтори. НЕ переходи дальше.

# Шаг 2: Типы (ОБЯЗАТЕЛЬНО)
mypy app tests
# Если ошибки — исправь и повтори. НЕ переходи дальше.

# Шаг 3: Тесты (ОБЯЗАТЕЛЬНО)
pytest -q
# Если падают — исправь. НЕ переходи дальше.

# Шаг 4: Компиляция (ОБЯЗАТЕЛЬНО)
python -m compileall -q app tests scripts

# Шаг 5: Лог (ОБЯЗАТЕЛЬНО)
# Добавь запись в docs/progress_report.md:
#   Дата | Что сделано | Какие файлы изменены

# Шаг 6: Коммит (Conventional Commits)
# feat: / fix: / refactor: / test: / docs: / chore:
```

## Процедура: добавление новой фичи

Порядок разработки — от центра архитектуры наружу:

1. **Domain** — создай/измени сущности, value objects, бизнес-правила.
2. **Infrastructure** — создай/измени SQLAlchemy модели, репозитории.
3. **Миграция БД** — если модель изменилась:
   ```bash
   alembic revision --autogenerate -m "описание"
   # ПРОВЕРЬ ФАЙЛ МИГРАЦИИ ВРУЧНУЮ!
   alembic upgrade head
   ```
4. **Application** — создай/измени use cases, application services.
5. **UI** — создай/измени PySide6 виджеты, диалоги.
6. **Тесты** — напиши тесты на КАЖДЫЙ затронутый слой.
7. **Quality gate** — полный цикл (см. выше).
8. **Лог** — обнови `docs/progress_report.md`.

## Процедура: исправление бага

1. **Тест-first**: напиши тест, воспроизводящий баг (он ДОЛЖЕН падать).
2. **Исправь баг** (тест должен проходить).
3. **Регрессия**: убедись, что остальные тесты не упали.
4. **Quality gate** → лог → коммит.

## Процедура: рефакторинг

1. Убедись, что есть тесты на рефакторируемый код.
2. Если тестов нет → **сначала напиши и закоммить тесты отдельным коммитом**.
3. Рефакторь малыми шагами. После каждого шага — `pytest -q`.
4. Quality gate → лог → коммит.

## Стандарты кода

### PySide6
- Виджеты через наследование QWidget / QDialog / QMainWindow.
- Layout-менеджеры обязательны — **НИКАКОГО абсолютного позиционирования**.
- Стили через QSS (resources/) — **НИКАКОГО inline стиля в Python**.
- Длинные операции → QThread / QRunnable — **НИКОГДА в основном потоке**.
- Данные из БД → QTableView + QAbstractTableModel.
- **НЕ ИСПОЛЬЗУЙ QML** — проект целиком на виджетах.

### SQLAlchemy 2
- Mapped[] аннотации (НЕ Column()).
- DeclarativeBase.
- relationship() с back_populates.
- Session через DI — **НЕ глобальные переменные**.
- Параметризованные запросы — **НИКАКОГО f-string в SQL**.

### Тесты (pytest)
- SQLite :memory: для тестов БД (мгновенный, настоящий SQL).
- **НИКОГДА mock для SQLAlchemy** — реальный :memory: лучше.
- **НИКОГДА mock для PySide6** если можно создать реальный виджет.
- Фикстуры в conftest.py.
- Именование: `test_<что>_<сценарий>`.

### Python
- Type annotations на ВСЕ (mypy strict).
- Docstrings на публичные API.
- snake_case / PascalCase.
- logging вместо print().
- Никаких `# type: ignore` без комментария.
- Никаких хардкод-путей — только через config.py / ENV.
