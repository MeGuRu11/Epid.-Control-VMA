# Spec-First: спецификация перед кодом

## Принцип

**Ни одна строка кода не пишется, пока фича полностью не описана.**
Спецификация — это ТЗ для тебя как AI-агента. Чем конкретнее описаны модели,
сервисы, UI — тем автономнее ты работаешь и меньше ошибок допускаешь.

## Когда активируется этот скилл

- Пользователь просит реализовать новую фичу
- Пользователь описывает новый модуль или экран
- Задача затрагивает более 3 файлов
- Нужно добавить новую таблицу/модель в БД

## Процедура

### Шаг 1: Написать спецификацию (ПЕРЕД кодом)

Создай файл `docs/specs/SPEC_<название>.md` по шаблону ниже.
Покажи пользователю и дождись подтверждения.

### Шаг 2: Утверждение

Пользователь должен сказать "ок", "да", "утверждаю", "делай" — только после этого пиши код.
Если пользователь вносит правки — обнови спецификацию и покажи снова.

### Шаг 3: Реализация по спецификации

Реализуй строго по утверждённой спецификации.
Если в процессе обнаруживаешь, что спека неполная — обнови её и сообщи пользователю.

### Шаг 4: Сверка

После реализации сверь код со спецификацией:
- Все user stories реализованы?
- Все edge cases обработаны?
- Тесты покрывают сценарии из спеки?

## Антипаттерны (НЕ делай так)

- ❌ Начинать писать код без спецификации
- ❌ Писать абстрактную спеку ("добавить продвинутую аналитику")
- ❌ Спека без edge cases
- ❌ Спека без описания модели данных
- ❌ Реализовывать то, что не утверждено пользователем

## Правило конкретики

Вместо: "Модуль отчётов с продвинутыми возможностями"
Пиши: "Экран аналитики: таблица проб за период с фильтрами по подразделению,
материалу и микроорганизму. Группировка по месяцам. Экспорт в Excel через
openpyxl. Данные из AnalyticsRepository.get_samples_by_period()."
