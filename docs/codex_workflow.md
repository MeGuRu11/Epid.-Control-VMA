# Codex Workflow

Этот документ добавляет лёгкий контур для долгих задач Codex без замены `.agents`, `AGENTS.md` и существующих quality gates.

## Что это решает

- даёт отдельный task-state для сложной задачи, который не смешивается с общим `docs/session_handoff.md`;
- делает checkpoint/resume предсказуемыми, если задача переживает паузу или смену сессии;
- фиксирует план, validation ledger и следующий шаг в одном месте;
- не создаёт вторую систему памяти поверх `docs/context.md`, `docs/progress_report.md` и `docs/session_handoff.md`.

## Когда использовать

Используйте этот контур, если задача:

- занимает больше одного крупного шага;
- затрагивает несколько модулей или слоёв;
- требует исследования перед правками;
- может быть прервана и продолжена позже;
- требует отдельного списка проверок и критериев готовности.

Для маленьких правок достаточно обычного цикла из `AGENTS.md`.

## Быстрый старт

1. Создайте task-файл:

```powershell
python scripts/codex_task.py new "Усилить workflow Codex"
```

2. Заполните в новом файле:
- `Цель и границы`
- `Контекст`
- `План`
- `Resume Point`

3. Только после этого переходите к правкам.

4. Перед каждым крупным шагом обновляйте:
- `Checkpoints`
- `Validation Ledger`
- `Resume Point`

5. После завершения синхронизируйте итог с:
- `docs/progress_report.md`
- `docs/session_handoff.md`

## Команды

Создать task-файл:

```powershell
python scripts/codex_task.py new "Краткое название задачи"
```

Создать task-файл с явным slug:

```powershell
python scripts/codex_task.py new "Краткое название задачи" --slug codex-workflow
```

Посмотреть последние task-файлы:

```powershell
python scripts/codex_task.py list
```

## Минимальный цикл long-task

### 1. Intake

- Прочитать `AGENTS.md`, `docs/context.md`, хвост `docs/progress_report.md`, `docs/session_handoff.md`.
- Если задача проектная, подключить релевантный skill из `.agents/skills/`.
- Создать task-файл и зафиксировать границы.

### 2. Plan

- Разбить задачу на 3-5 шагов максимум.
- Отдельно зафиксировать:
  - что считается готовым;
  - что нельзя менять;
  - какие quality gates обязательны;
  - какие ручные проверки нужны.

### 3. Execute

- Работать малыми шагами.
- Перед серией правок или сменой контекста обновлять `Checkpoints`.
- Не переносить промежуточные заметки в `docs/session_handoff.md`, пока задача не подходит к паузе или завершению.

### 4. Verify

- Закрыть `Validation Ledger`.
- Сверить фактический результат с `Планом`.
- Если есть остаточный риск, зафиксировать его в `Resume Point` и `docs/session_handoff.md`.

### 5. Handoff

- Обновить `docs/progress_report.md`.
- Обновить `docs/session_handoff.md`.
- Если задача незавершена, оставить в task-файле конкретный следующий шаг, а не общий комментарий.

## Что намеренно НЕ добавлено

- отдельная `.omx`-память;
- автоматический runtime/daemon;
- мульти-агентные роли по умолчанию;
- hook-автоматика, завязанная на нестабильное поведение Codex CLI на Windows.

Это сделано специально: текущий репозиторий уже хорошо покрыт `AGENTS.md`, `.agents`, `progress_report`, `session_handoff` и quality gates. Здесь нужен не второй framework, а тонкий слой поверх них.
