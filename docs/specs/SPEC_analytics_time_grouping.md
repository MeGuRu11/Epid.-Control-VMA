# Спецификация: группировка временной шкалы аналитики

Дата: 2026-05-04
Автор: Codex + пользователь
Статус: утверждена пользовательским ТЗ

## Описание

В разделе `Аналитика` добавляется режим группировки временного ряда тренда: `Авто`, `Дни`, `Недели`, `Месяцы`. Функция нужна для читаемого просмотра длинных периодов без потери данных и без смешивания агрегации с компонентом отрисовки графика.

## User Stories

1. Как пользователь, я хочу выбрать детализацию тренда по дням, неделям или месяцам, чтобы анализировать короткие и длинные периоды в подходящем масштабе.
2. Как пользователь, я хочу режим `Авто`, чтобы приложение само выбирало детализацию по длине периода.
3. Как пользователь, я хочу, чтобы смена группировки не сбрасывала остальные фильтры аналитики.

## UI (PySide6)

### Экраны / виджеты

| Виджет | Тип | Описание |
|--------|-----|----------|
| `compare_period` | `QComboBox` | Отдельный существующий контрол `Период сравнения` в блоке `Сводка`: `Неделя`, `Месяц` |
| `time_grouping` | `QComboBox` | Отдельный видимый контрол `Группировка` в блоке `Сводка` рядом с `Период сравнения`: `Авто`, `Дни`, `Недели`, `Месяцы` |
| `TrendChart` | `QWidget` | Получает уже подготовленные labels/values и только отрисовывает график |

`Период сравнения` и `Группировка` не взаимозаменяемы: первый управляет окном сравнения текущего периода с предыдущим, второй управляет агрегацией временной шкалы графика тренда.

### Состояния UI

- Загрузка: существующее поведение аналитики сохраняется.
- Пусто: пустой ряд не падает, график очищает ticks.
- Ошибка: существующая обработка ошибок аналитики сохраняется.

## Бизнес-логика

### Правила группировки

1. `Дни`: дневная детализация без агрегации, значения одинаковых дат суммируются.
2. `Недели`: группировка по ISO-неделям, label формата `YYYY-Www`, totals/positives суммируются.
3. `Месяцы`: группировка по календарным месяцам, label формата `MM.YYYY`, totals/positives суммируются.
4. `Авто`: до 31 дня включительно — `Дни`; 32-180 дней — `Недели`; больше 180 дней — `Месяцы`.

### Формулы / вычисления

- Для каждой группы `positive_share = positives / total * 100`, если `total > 0`, иначе `0.0`.
- Агрегация выполняется до передачи данных в `TrendChart`.
- Прореживание подписей X-оси через `build_axis_ticks(...)` сохраняется и не удаляет данные.

## Edge Cases

1. Пустой ряд данных.
2. Один день.
3. Периоды ровно 31, 32, 180 и 181 день.
4. Даты на границе месяца.
5. ISO-недели на границе года.
6. Повторяющиеся даты в исходных rows.
7. Короткий период при ручном режиме `Дни`.

## Безопасность

- Проверка прав доступа: существующая логика аналитики не меняется.
- Валидация ввода: значение группировки берётся из фиксированного `QComboBox`.
- Аудит-лог: не требуется, операция не мутирует данные.
- Персональные данные не логируются.

## Тесты

### Что покрыть тестами

1. Unit: `resolve_time_grouping` для порогов авто-режима.
2. Unit: `group_trend_rows` для дней, ISO-недель, месяцев, границ года и пустого ряда.
3. Unit UI: combobox группировки существует, default `Авто`, смена значения обновляет аналитику.
4. Regression: `TrendChart` продолжает получать полный набор labels/values, а `build_axis_ticks(...)` прореживает только подписи.
5. Integration: существующие analytics service date boundary/query tests не должны измениться.
