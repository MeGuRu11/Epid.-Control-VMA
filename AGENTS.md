## 🔒 БЕЗОПАСНОСТЬ (медицинские данные!)

Проект обрабатывает персональные данные пациентов. Это накладывает повышенные требования.

### Обязательные правила безопасности

1. **Пароли:** ТОЛЬКО bcrypt/argon2. НИКОГДА md5/sha1/sha256/plain text.
2. **SQL:** ТОЛЬКО параметризованные запросы. НИКОГДА f-string в DML (SELECT/INSERT/UPDATE/DELETE).
3. **Ввод:** ВСЕ данные из UI валидируются перед отправкой в application-сервис.
4. **Аудит:** Каждая мутирующая операция (создание/изменение/удаление) логируется с actor_id.
5. **Роли:** Каждая операция проверяет права доступа пользователя.
6. **Данные пациентов:** НИКОГДА не логировать ФИО, паспорт, СНИЛС, полис в консоль/лог-файл. Только ID.
7. **Секреты:** НИКОГДА не хардкодить пароли/ключи. Только через ENV или config.py.
8. **Транзакции:** Все операции с БД через session_scope() (context manager). При ошибке — rollback.

### Перед каждым релизом

Запусти security-review: `$security-review` — скилл проведёт полный аудит.

### При добавлении нового ввода от пользователя

1. Валидируй тип и длину в UI (форменная валидация).
2. Валидируй бизнес-правила в Application/Domain.
3. Используй параметризованные запросы в Infrastructure.

## ⚠️ КРИТИЧЕСКИ ВАЖНЫЕ ПРАВИЛА

### Язык
- Общение, комментарии к коду, коммиты, документация — **ТОЛЬКО на русском языке**.
- Имена переменных, функций, классов, модулей — на английском (Python-конвенция).

### Обязательный цикл после КАЖДОГО изменения кода
1. **Линтер:** `ruff check app tests` → если ошибки, исправь ДО перехода дальше.
2. **Типы:** `mypy app tests` → если ошибки, исправь ДО перехода дальше.
3. **Тесты:** `pytest -q` → если падают, исправь ДО перехода дальше.
4. **Компиляция:** `python -m compileall -q app tests scripts`
5. **Лог:** добавь запись в `docs/progress_report.md` — что сделано, какие файлы изменены.
6. **Коммит:** оформи по Conventional Commits.

**НИКОГДА не пропускай шаги 1-4.** Если нет времени — лучше сделай меньше, но качественно.

### Запрещено
- Пушить код, не прошедший quality gates.
- Менять структуру БД без Alembic-миграции.
- Удалять или изменять существующие тесты без объяснения причины.
- Использовать `# type: ignore` без комментария, почему.
- Хардкодить пути — используй `app/config.py` и ENV-переменные.
- Писать GUI-логику в слое данных или бизнес-логику в UI-слое.
- Создавать файлы вне структуры проекта.
- Ломать обратную совместимость БД без миграции.

## О проекте

**Epid Control** — десктопное приложение для эпидемиологического контроля и микробиологических рабочих процессов в ЭМЗ.

### Стек
| Компонент | Технология |
|-----------|-----------|
| Язык | Python 3.11+ (рекомендуется 3.12) |
| UI | PySide6 (виджеты, НЕ QML) |
| ORM | SQLAlchemy 2 (Mapped[] стиль) |
| БД | SQLite (файловая) |
| Миграции | Alembic |
| Линтер | ruff |
| Типы | mypy (strict) |
| Тесты | pytest |
| CI | GitHub Actions |
| Сборка | PyInstaller |
| Платформа | Windows-first, cross-platform ready |

### Архитектура: Clean Architecture / DDD

```
UI Layer (PySide6 виджеты, диалоги, формы)
    ↓ только через Application Layer
Application Layer (use cases, сервисы приложения)
    ↓ через интерфейсы/протоколы
Domain Layer (сущности, value objects, бизнес-правила)
    ↑ Infrastructure реализует интерфейсы Domain
Infrastructure Layer (SQLAlchemy модели, репозитории, I/O)
```

**Правила зависимостей:**
- UI → Application ✅
- UI → Domain ❌ (только через Application)
- UI → Infrastructure ❌ (никогда напрямую)
- Application → Domain ✅
- Application → Infrastructure ✅ (через интерфейсы)
- Domain → ничего ✅ (чистый слой, без внешних зависимостей)
- Infrastructure → Domain ✅ (реализует интерфейсы)

## Рабочие процедуры

### Перед началом любой задачи
1. Прочитай `docs/session_handoff.md` — контекст предыдущей сессии.
2. Прочитай `docs/context.md` — пойми текущий роадмап и состояние.
3. Прочитай `docs/progress_report.md` — пойми, что было сделано недавно.
4. Определи, какие слои архитектуры затронуты.
5. Если задача затрагивает скилл из `.agents/skills/` — прочитай его `SKILL.md`.

### Добавление новой фичи
1. **СНАЧАЛА** напиши спецификацию: `$spec-first` → `docs/specs/SPEC_<название>.md`.
2. Покажи пользователю, дождись утверждения.
3. Начни с Domain Layer (сущности, бизнес-правила).
4. Затем Infrastructure (модели SQLAlchemy, репозитории).
5. Если изменилась модель БД → создай Alembic-миграцию (см. ниже).
6. Затем Application (use cases, сервисы).
7. В последнюю очередь — UI (PySide6 виджеты).
8. Напиши тесты на **КАЖДЫЙ затронутый слой**.
9. Пройди полный quality gate цикл.
10. Обнови `docs/progress_report.md`.
11. Коммит по Conventional Commits.

### Изменение модели БД
1. Измени SQLAlchemy-модель в Infrastructure.
2. Создай миграцию: `alembic revision --autogenerate -m "краткое описание на русском"`
3. **ОБЯЗАТЕЛЬНО проверь сгенерированный файл миграции** — autogenerate не идеален, может пропустить rename, default values, индексы.
4. Примени: `alembic upgrade head`
5. Проверь тесты: `pytest -q`
6. Если нужен seed: обнови `scripts/seed_references.py`.

### Исправление бага
1. **Сначала** напиши тест, воспроизводящий баг (он должен падать — red).
2. Исправь баг (тест должен проходить — green).
3. Убедись, что ничего не сломалось: полный `pytest -q`.
4. Рефакторь если нужно.
5. Quality gate цикл.
6. Обнови `docs/progress_report.md`.

### Рефакторинг
1. Убедись, что есть тесты, покрывающие рефакторируемый код.
2. Если тестов нет — **сначала напиши тесты**, закоммить их отдельно.
3. Рефакторь малыми шагами, прогоняя тесты после каждого шага.
4. Quality gate цикл после завершения.

## Стандарты кода

### Python общее
- Type annotations на ВСЕ функции, методы и переменные уровня класса.
- Docstrings на публичные классы и методы (формат: краткое описание, Args, Returns, Raises).
- snake_case для функций/переменных, PascalCase для классов.
- Импорты: сортировка через ruff (isort-совместимая).
- Никаких `print()` в продакшн-коде — используй logging.

### PySide6 виджеты
- Наследуйся от QWidget / QDialog / QMainWindow.
- Layout-менеджеры обязательны (QVBoxLayout, QHBoxLayout, QGridLayout) — никакого абсолютного позиционирования.
- Сигналы подключай через `.connect()`.
- Стилизация через QSS (файлы в resources/) — никакого inline-стиля в Python-коде.
- **НЕ используй QML** — проект целиком на виджетах.
- Длинные операции — в QThread или QRunnable, **НИКОГДА в основном потоке** (UI замёрзнет).
- Для показа данных из БД используй QTableView + QAbstractTableModel.

### SQLAlchemy 2
- Mapped[] аннотации (новый стиль, не Column()).
- DeclarativeBase для базового класса.
- relationship() с явным back_populates.
- Session через dependency injection — НЕ глобальные переменные.
- Параметризованные запросы — НИКАКОГО форматирования строк в SQL.

### Тесты (pytest)
- Фикстуры для БД: SQLite :memory: через tmp_path или фикстуру engine.
- Каждый тест изолирован — своя сессия, свои данные, rollback после теста.
- Именование: `test_<что>_<сценарий>` (например `test_create_sample_with_empty_name_raises_error`).
- conftest.py для общих фикстур.
- **НИКОГДА не используй моки для SQLAlchemy** — используй реальный SQLite :memory:, он мгновенный.
- **НИКОГДА не используй моки для PySide6 виджетов** если можно создать реальный виджет в тесте.

### Коммиты (Conventional Commits)
```
feat: добавлена форма ввода проб
fix: исправлен расчёт сроков хранения
refactor: вынесен сервис валидации в отдельный модуль
test: покрыты edge-cases для импорта данных
docs: обновлён user_guide.md
chore: обновлены зависимости
build: обновлён EpidControl.spec
ci: добавлен шаг проверки миграций
```
